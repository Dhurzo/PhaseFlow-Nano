# PhaseFlow Nano: Multi-Agent System for OpenCode

System of 8 specialized agents that work with files as source of truth, optimized for local models with reduced context windows.

## Available Agents

| Agent | Type | Role | When to use |
|--------|------|-----|---------------|
| `phaseflow-explorer` | 🔍 Explorer | Analyzes existing codebases (stack, patterns, debt) | Before planning if the project ALREADY has code |
| `phaseflow-planner` | 📋 Planner | Splits work into independent, self-contained phases. **Analyzes existing code to map relevant files into each phase. Asks clarifying questions when requirements are vague.** | When the user asks to plan, split into stages |
| `phaseflow-builder` | ⚙️ Executor | Executes backend/logic phases sequentially | API, DB, business logic, configuration phases |
| `phaseflow-builder-visual` | 🎨 UI Executor | Executes frontend/design phases (⚠️ untested) | Component, CSS, layout, accessibility phases |
| `phaseflow-refiner` | ✏️ Refiner | Improves phase files by identifying ambiguities, researching technologies, and clarifying with the user | **Manually**, after planning, when a phase has vague prompts |
| `phaseflow-reviewer` | 🔬 Auditor | Reviews deliverables for bugs and issues | After EVERY completed phase, before continuing |
| `phaseflow-orchestrator` | 🎯 Orchestrator | Runs the full pipeline unattended (builder → reviewer loop, auto-retries REQUIRES_FIX up to 3×) | When the user wants ALL phases executed without manual intervention |
| `phaseflow-doctor` | 🩺 Diagnostician | Inspects project health — validates phase structure, states, and file consistency | When phases get stuck, states are inconsistent, or before/after planning |

## Philosophy

- **State lives in files, not in the conversation.**
- Each phase is self-contained (does not depend on prior context).
- Context is reset between phases.
- Progress is persisted in `plan.md`.

## Complete Workflow

### Greenfield project (from scratch)
```
1. phaseflow-planner  → generates plan.md + phases/*.md
   └─ optional → /phaseflow-refine phase-N.md  ← refine vague phases
2. phaseflow-builder → executes phase 1
   ├─ checkpoints after each task (CHECKPOINT.md)
   └─ writes CONTEXT.md for next phase
3. phaseflow-reviewer → audits phase 1
4. Repeat 2-3 until complete
   └─ if REQUIRES_FIX → orchestrator auto-retries (up to 3×)
```

### Existing project (with code)
```
0. phaseflow-explorer → generates explore-report.md
   └─ includes ## Project Snapshot for planner
1. phaseflow-planner  → plans using the explorer's report;
                     analyzes the codebase to map relevant
                     existing files into each phase via
                     `## Related Files` section;
                     copies ## Project Snapshot from explorer
   └─ optional → /phaseflow-refine phase-N.md  ← refine vague phases
2. phaseflow-builder → reads the phase, reads Related Files
                     for context, reads CONTEXT.md from
                     dependency phases automatically,
                     executes phase 1
3. phaseflow-reviewer → audits phase 1
4. Repeat 2-3 until complete
   └─ if REQUIRES_FIX → orchestrator auto-retries (up to 3×)
```

### Frontend/UI phase
```
X. phaseflow-builder-visual → executes the phase with design sensibility
X+1. phaseflow-reviewer → audits (accessibility, responsive, visual quality)
```

### Automated (orchestrator mode)
```
1. phaseflow-orchestrator → reads plan.md
2. Invokes phaseflow-builder (or phaseflow-builder-visual) for PENDING/REQUIRES_FIX phase
3. Invokes phaseflow-reviewer for COMPLETED phase
4. Repeats 2-3 until all phases are REVIEWED, BLOCKED, or ERROR
```

## File Structure

```
project/
├── plan.md              ← Master file with phase table and states
├── DECISIONS.md         ← Accumulated cross-phase technical decisions
├── explore-report.md    ← Generated by phaseflow-explorer
├── phases/
│   ├── phase-1.md       ← Self-contained phase (includes Related Files)
│   ├── phase-2.md       ← Self-contained phase (includes Related Files)
│   └── phase-N.md
├── outputs/
│   ├── phase-1/
│   │   ├── .phase        ← State: one word (pending/in_progress/completed/…)
│   │   ├── [generated files]
│   │   ├── CONTEXT.md   ← Generated by builder (structured context for next phase)
│   │   ├── CHECKPOINT.md ← Generated by builder (resume data, deleted on completion)
│   │   ├── REVIEW.md    ← Generated by phaseflow-reviewer
│   │   ├── SUMMARY.md   ← Generated by builder (includes TL;DR and Reviewer Notes)
│   │   └── remaining-tasks.md ← Generated by /phaseflow-stop (pause state for resume)
│   ├── phase-2/
│   └── FINAL-REPORT.md  ← Generated by orchestrator at the end
└── logs/                ← Execution logs
```

## Phase States — Two Sources of Truth

### 1. `.phase` files (programmatic state — lowercase, no newline)

Each `outputs/phase-X/.phase` contains one word:
- `pending` — ready to execute
- `in_progress` — currently executing, or paused/interrupted mid-execution (check remaining-tasks.md / CHECKPOINT.md for resume point)
- `completed` — finished, awaiting review
- `reviewed` — passed review ✅
- `requires_fix` — reviewer found issues
- `blocked` — missing information
- `error` — unrecoverable failure

### 2. `plan.md` table (human-readable state — uppercase)

- `PENDING` → `IN_PROGRESS` → `COMPLETED` → `REVIEWED` (standard flow)
- `COMPLETED` → `REQUIRES_FIX` → `COMPLETED` → `REVIEWED` (fix cycle)
- Any state → `BLOCKED` or `ERROR` (terminal)

> **PAUSED has been removed.** Pausing uses the `in_progress` state with `remaining-tasks.md` as the resume signal. No separate state needed.

Each phase in `plan.md` also has a **Type**: `Backend/Logic` (handled by `phaseflow-builder`) or `Visual/Frontend` (handled by `phaseflow-builder-visual`). The table also tracks **Key Outputs** and **Key Decisions** — filled by the builder after completion so any agent can scan the plan and understand what each phase produced and decided. Every phase file includes a `## Project Snapshot` section — either inline (legacy) or `> See \`plan.md\` → \`## Project Snapshot\`` (preferred).

## When to Use Each Agent

### phaseflow-explorer
- The project ALREADY has code
- Need to "explore the codebase" before planning
- User asks to "analyze the existing project"

### phaseflow-planner
- User asks to "plan X"
- User describes a large project
- Need to split work into stages
- Need to reorganize existing phases
- When the project has existing code, the planner will analyze relevant files and map them into each phase via `## Related Files`
- **New**: When requirements are vague, the planner will ask clarifying questions before generating the plan

### phaseflow-builder
- User asks to "execute the plan" or "continue"
- There are pending `Backend/Logic` phases in `plan.md`
- The phase is backend, logic, DB, or configuration
- Need to make progress on the build
- The builder reads `## Related Files` from the phase for context before executing
- **Refuses `Visual/Frontend` phases** — use `phaseflow-builder-visual` instead

### phaseflow-builder-visual
- The pending phase is UI/Frontend type
- User asks to "execute the design phase"
- The phase includes visual components, CSS, animations
- **⚠️ Untested** — this agent has not been validated in real projects. Review output carefully and expect potential issues.

### phaseflow-refiner
- **Manually**, after planning, when a phase has vague prompts
- The phase tasks are too abstract or have unclear deliverables
- Missing edge cases, error handling, or constraints in a phase
- Need to make a phase more concrete before the builder executes it
- **New**: Can research technologies (web search) to fill in best practices, versions, and patterns without asking the user

### phaseflow-reviewer
- A phase has just completed (invoke automatically)
- User asks to "review the code" or "audit the phase"
- Before continuing to the next phase

### phaseflow-doctor
- Phases are stuck in non-terminal states
- States seem inconsistent (e.g., phase marked COMPLETED but outputs/ missing)
- Before or after planning, to validate file structure
- User asks to "check project health", "diagnose", "validate phases", or "/phaseflow-doctor"

### phaseflow-orchestrator
- User asks to "run everything automatically", "/phaseflow-orchestrate", or "execute all phases"
- There's a full plan with multiple PENDING phases
- The user wants a fire-and-forget experience
- Alternative to manually invoking builder → reviewer → builder → reviewer...
- The orchestrator reads plan.md, invokes builders and reviewers as sub-agents (fresh context each time), and stops when all phases reach a terminal state
- At the end, reads all `outputs/phase-*/SUMMARY.md` and displays a concise per-phase recap on screen + saves `outputs/FINAL-REPORT.md`

## Quick Commands

### `/phaseflow-status`
- Shows PhaseFlow project status at a glance
- Reads `plan.md` and displays: project name/goal, phase table with states, and which phase to execute next
- Safe to use in any session, zero side effects (read-only)

### `/phaseflow-refine <phase-N.md>`
- Refines a phase file by identifying ambiguities, missing details, and gray areas
- Asks clarifying questions and rewrites the phase with concrete, actionable prompts
- **Manual, optional step** — use after planning when a phase feels vague

### `/phaseflow-doctor`
- Diagnoses project health — validates phase structure, states, and file consistency
- Checks: orphaned files, missing phase files, inconsistent states, corrupted CONTEXT.md
- Safe to use in any session, zero side effects (read-only)

### `/phaseflow-stop`
- Pauses the currently executing phase (in_progress state)
- Writes `outputs/phase-X/remaining-tasks.md` with current progress for clean resume
- Keeps state as `in_progress` (no separate PAUSED state — remaining-tasks.md is the resume signal)
- Safe to use after an agent interruption or context overflow

## Rules for the Primary Agent

1. **Explore first** if the project has code → invoke `phaseflow-explorer` via `inherit-task` (preferred) or `task` (fallback). Do NOT explore directly.
2. **Plan** → invoke `phaseflow-planner` via `inherit-task` (preferred) or `task` (fallback). Do NOT plan directly.
3. **Execute phase** → invoke `phaseflow-builder` (or `phaseflow-builder-visual` if UI) via `inherit-task` (preferred) or `task` (fallback). Do NOT execute directly.
4. **Review** → invoke `phaseflow-reviewer` via `inherit-task` (preferred) or `task` (fallback) AFTER the builder completes. Do NOT review directly.
5. **Stop after ONE sub-agent invocation.** After invoking a planner, builder, reviewer, or explorer, STOP. Report the result to the user and tell them what the next step is. The user will trigger the next invocation manually. Do NOT invoke the orchestrator or any additional agent automatically.
6. NEVER try to plan and execute in the same conversation
7. Context resets between builder/reviewer invocations
8. For unattended execution, the user will invoke `/phaseflow-orchestrate` manually. Do NOT invoke the orchestrator on your own initiative — only the user's explicit `/phaseflow-orchestrate` command triggers it.
